# SchemeStealer — Data & Database

## The asset

The ground-truth database — **1,300+ paints (≈1,216 base + 96 washes)** physically
photographed as swatches, with measured LAB — is the product's primary competitive asset.
Every data change is a change to the moat: be conservative, provenance-aware, and
deduplicate aggressively.

**Brands (six, fixed):** Citadel, Vallejo (Model Color / Game Color / Model Wash / Metal
Color), Army Painter (Warpaints Fanatic / WPH / speedpaint lines as tagged), AK 3rd Gen,
Pro Acryl, Two Thin Coats. **Scale75 is dropped** — entries may linger in legacy files;
never add new Scale75 data, never surface it in the UI, and remove it when touching a file
that contains it (with Cam's approval).

## Files and their roles

| File | Role | Edit policy |
|---|---|---|
| `python-api/paints_measured.json` | Canonical ground truth (measured_hex, measured_lab, opacity, reference_families) | Source of truth; edit only via approved data prompts |
| `schemestealer-react/lib/data/paints_groundtruth.json` | Frontend ground-truth mirror | Generated/synced from canonical — do not hand-diverge |
| `schemestealer-react/lib/paintDatabase.ts` | Frontend runtime DB (incl. rgb, lab, family, transparency) | Generated — regenerate, don't hand-edit |
| `python-api/color_anchors.json` + `lib/colorAnchors.ts` | Classifier exemplars | Built by `scripts/build_color_anchors.py`; NEVER hand-edit either |
| `python-api/recipes.json` | Highlight/shade partner graph (`rel`, `source: algorithmic`, `confidence`) | Regenerated by recipe tooling |
| `schemestealer-react/lib/data/conversions.json` | Pre-computed cross-brand ΔE matches for SEO pages | Generated by the Python batch script only |
| `python-api/swatch_resample.json` | Swatch re-measurement audit (drift_de00 per paint) | Measurement tooling output |

## Schema (canonical record)

```json
{
  "paint_id": "brand-kebab-name",          // stable, unique, never recycled
  "name": "…", "brand": "…", "range": "…",
  "hex": "#RRGGBB",                        // legacy/marketing hex
  "measured_hex": "#rrggbb",               // from swatch photo — preferred
  "measured_lab": [L, a, b],               // D65; THE matching value
  "color_source": "measured" | "legacy",
  "hex_source": "measured_swatch" | "legacy",
  "category": "base" | "wash",
  "finish": "matte" | "flat" | "satin" | "glossy",
  "transparency": 0.0–1.0,  "opacity": 0.0–1.0,  "opacity_rating": 0–3,
  "color_family": "lowercase family",
  "reference_families": ["…"],  "reference_families_source": "derived",
  "metallic": true|false,  "matchable": true|false,  "discontinued": true|false,
  "aliases": [], "tags": ["measured", "family:…"]
}
```

Rules:
- **Matching always uses `measured_lab`** when present (the engine sets
  `self.lab = measured_lab` in `compute_properties`); hex is display/legacy only.
- Family values are lowercase in data; Capitalised only at display.
- Known family set includes: black, white, grey, bone, brown, red, orange, yellow, green,
  cyan, blue, purple, magenta, pink, gold, silver, bronze. Do not invent families —
  family changes are classifier changes (colour-science skill, fixture-first).
- Washes: `category: "wash"`, `transparency: 1`. The washes file was historically messy
  (malformed placeholders, concatenated arrays) — when ingesting new wash data, validate
  JSON structure, merge arrays, and dedupe against existing measured washes by
  `paint_id` and by name+brand.

## ΔE quality bands (consistent everywhere)

Computed with **CIEDE2000** by the Python engine only:
`perfect` (≲2) · `close` (≲5) · `fair` (≲10) · `distant` (>10). If exact thresholds in
the live batch script differ, the script is authoritative — read it, don't assume.

## Programmatic SEO pipeline

Flow: Python batch script → static `conversions.json` → Next.js SSG pages at
`/paints/[brand]/[slug]` and `/convert/[fromSlug]-to-[brand]`.

Hard rules:
1. **All ΔE computation happens in Python at build time.** No colour maths reimplemented
   in TypeScript for these pages (core invariant 9).
2. Slugs derive from `paint_id`; keep them stable — they become indexed URLs.
3. Each conversion entry: source paint + per-brand top-5 matches with `delta_e` and
   `band`. Never fabricate a match; if a brand has no entry within range, omit it.
4. Page copy: British English, no swatch-source attribution, honest about `distant` matches.

## Step-by-step: any data change

1. Identify which files the change touches using the table above; note which are
   **generated** (regenerate, never hand-edit).
2. Validate incoming data structurally first: parse the JSON, check for concatenated
   arrays, placeholder junk, duplicate `paint_id`s, missing `measured_lab`.
3. Dedupe: by `paint_id`, then by (brand, normalised name). Prefer the record with
   `color_source: "measured"`; keep the alias on the survivor.
4. Apply the change to the canonical file (`paints_measured.json`) via an approved prompt.
5. Regenerate downstream artefacts in order: frontend DB/groundtruth mirror → anchors
   (`build_color_anchors.py`, only if families/measured values changed) →
   `recipes.json` if partner-relevant → `conversions.json` if matching-relevant.
6. Run the parity suites (Python + TS golden fixtures) if anchors or families changed;
   spot-check a few regenerated conversion entries against manual CIEDE2000.
7. Self-check: counts sane (≈1,300 total)? No Scale75 added? No swatch-source
   attribution? Stable `paint_id`s? British English in any generated copy?
